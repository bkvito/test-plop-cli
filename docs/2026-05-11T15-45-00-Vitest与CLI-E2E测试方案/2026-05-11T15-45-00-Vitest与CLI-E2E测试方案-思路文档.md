# Vitest 与 CLI E2E 测试方案思路文档

## 背景

当前 `scripts/verify-templates.mjs` 的校验方式和模板内容绑定较深。模板做小改动时，经常需要同步修改脚本断言，这会让测试更像手工清单，而不是稳定的质量保障。

## 思路

- 测试应优先验证生成器的稳定契约，而不是逐字验证模板内容
- `generateComponent()` 适合做集成测试，因为它已经封装了模板选择、变量替换、文件写入和冲突策略
- CLI 入口仍需要 E2E 测试，因为参数解析、退出码和 stdout/stderr 都是用户真实体验的一部分

## 取舍

- 暂不引入快照测试，避免模板频繁调整时维护成本过高
- 暂不删除旧验证脚本，先让正式测试体系和现有验证命令并存
- 测试断言保留少量关键片段，用于确认变量替换和核心入口正确
