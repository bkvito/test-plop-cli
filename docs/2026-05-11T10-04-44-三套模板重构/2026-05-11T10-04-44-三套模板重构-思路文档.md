# 三套业务模板重构思路文档

## 1. 这次真正要解决的问题

补充需求之后，问题已经不是“把 `setup/normal` 改成三种业务模板”。

真正的问题是：

1. 业务模板层需要扩展为三种
2. Vue 写法层仍然保留两种
3. 这两个层次不能继续混用一个参数语义

也就是说，当前项目要从“单维模板选择”升级成“双维模板选择”。

## 2. 为什么之前会容易混淆

当前代码里，`templateType` 实际控制的是：

- `index.vue` 用 `setup`
- 或者 `index.vue` 用 `normal`

这会让人自然以为 `templateType` 就是“模板类型”。

但现在已经确认：

- `general` / `single-pick` / `multiple-pick` 才是业务模板类型
- `setup` / `normal` 只是 Vue 文件写法

因此，之前的困惑本质上来自“参数命名承载了错误的业务语义”。

## 3. 我对结构的重新判断

重构后最合理的抽象应该是：

### 第一层：业务模板目录

这一层决定：

- `typing.ts`
- `hook.ts`
- `data.ts`
- `index.scss`
- `index.ts`
- 以及 `index.vue` 的业务语义部分

### 第二层：Vue 风格分支

这一层只决定：

- `index.vue` 采用 `setup`
- 还是 `defineComponent`

所以目录结构不应该是：

```txt
setup/
normal/
```

而应该是：

```txt
general/
single-pick/
multiple-pick/
```

然后在每个业务目录下再放：

```txt
index.setup.vue.hbs
index.normal.vue.hbs
```

但这里要注意：

```txt
上面这两个名字只是模板源文件名，
最终生成到组件目录里的文件名仍然统一是 index.vue
```

## 4. 为什么这样拆最稳

因为这符合差异的主次关系。

### 主差异

主差异是业务模板不同。

这会影响整套模板资源的含义，所以它应该成为目录分层的第一优先级。

### 次差异

次差异是 `index.vue` 的 Vue 写法不同。

这只影响单个文件，因此应该成为业务目录下的次级分支，而不是顶层结构。
它影响的是“选哪份模板源文件”，不影响最终输出文件名。

## 5. 参数层面的思考

如果继续只保留一个 `templateType`，后面无非两种结果：

1. `templateType` 既可能传业务值，又可能传 Vue 风格值
2. 代码内部继续做大量兼容判断

这两种都不利于长期维护。

所以我更倾向于把参数拆开，例如：

- `templateType`
- `vueTemplateType`

这样能直接把概念表达清楚。

## 6. 兼容旧调用的思路

这里不能简单粗暴地完全切断旧逻辑，因为现有项目已经支持：

```bash
--templateType=setup
--templateType=normal
```

如果马上废弃，旧脚本会直接失效。

所以更合适的迁移方式是：

1. 新逻辑正式引入双维度参数
2. 旧逻辑临时兼容
3. 兼容时把旧值映射到：
   - 业务模板默认 `general`
   - Vue 风格沿用旧值

这个映射是合理的，因为旧调用本来就没有表达业务模板差异。

## 7. 对模板文件生成阶段的判断

你前面说“各个类型模板文件先给我生成，文件内容参考现有文件”，这意味着后续实施阶段的重点不是一次性把业务逻辑填满，而是先把骨架建立完整。

所以执行时更适合：

1. 先复用现有模板为基础
2. 再按三类业务语义做差异化改造
3. 最后再接回生成器逻辑

这样风险更低，也更容易验证。

## 8. 最终思路结论

这次最关键的思路不是“模板数量从 2 变成 3”，而是：

```txt
原来只有一个维度：
Vue 写法维度

现在变成两个维度：
业务模板维度 + Vue 写法维度
```

只要把这个抽象立住，后面的代码改造、目录设计、参数设计和测试方案都会自然顺下来。

补充一个最终落地规则：

```txt
模板源文件可以区分 setup / normal，
最终组件目录中只能生成统一的 index.vue
```
